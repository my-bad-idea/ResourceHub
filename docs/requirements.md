## 资源导航系统（ResourceHub）需求文档

### 1. 文档目的

本文件面向产品、研发与运维同学，简要说明 ResourceHub 作为通用资源管理产品的核心能力、使用场景和约束，帮助新成员在几分钟内理解“这个系统是做什么的”和“必须保证哪些行为不变”。

### 2. 系统概述

- **系统定位**：面向通用资源管理场景的应用，用来集中整理、访问和维护站点、工具、知识库及其他链接资源，不限定特定行业，也不限定组织类型或内容边界。
- **主要角色**：
  - 未登录访客：只能浏览公开且已启用的资源。
  - 普通用户：在访客能力基础上，可以创建/管理自己的资源，收藏、查看访问历史。
  - 管理员：除普通用户能力外，还负责用户、类别、标签、系统参数和邮件服务等全局配置。
- **核心价值**：
  - 统一资源库：把分散在各处的站点、工具、文档与知识条目统一到一个可检索、可维护的资源库中。
  - 权限可控：支持公开/私有资源、账号状态控制，仅授权用户可访问敏感链接。
  - 行为可回溯：记录收藏与访问历史，便于回顾和发现高频资源。
  - 配置集中：站点标题、副标题、注册策略和邮件服务统一通过后台配置。

### 3. 核心功能

#### 3.1 认证与账号

- 系统首次运行时，通过初始化向导创建首个管理员账号。
- 登录：
  - 用户名 + 密码登录。
  - 登录成功后返回 JWT，前端仅在 `sessionStorage` 中保存 token。
- 注册：
  - 是否允许开放注册由系统配置控制。
  - 注册时不需要用户自设密码，系统自动生成随机初始密码，并通过邮件（或 Mock 邮件预览）告知。
- 忘记密码与重置密码：
  - 用户可通过邮箱申请重置密码，系统生成一次性重置 token。
  - 在重置页面填写新密码后生效，旧 token 标记为已使用。
- 个人信息与密码修改：
  - 登录用户可修改自己的显示名称和邮箱。
  - 需提供当前密码后才能修改密码，且新旧密码不能相同。

#### 3.2 资源管理

- 资源作为系统的核心实体，代表一个可访问的链接（工具、系统、文档等）。
- 支持的主要字段：名称、URL、类别、描述、Logo、标签、公开/私有、启用状态、访问计数等。
- 基本能力：
  - 新增、编辑、删除资源。
  - 配置资源为公开或私有：
    - 公开：任何用户都可以看到（未登录用户只看公开且启用的资源）。
    - 私有：仅资源 owner（创建人）和管理员可见。
  - 可通过类别、标签、快速访问（收藏/历史/我创建的）以及关键词搜索进行组合筛选。
- 展示视图：
  - 卡片视图：以卡片形式展示 Logo、名称、类别、标签、描述和私有标识。
  - 列表视图：以表格行形式展示，便于快速扫描和管理。
  - 时间轴视图：按创建时间分组展示资源，方便按时间线回顾。

#### 3.3 类别与标签

- **类别（Category）**：
  - 用于将资源分门别类，如“开发工具”“设计资源”“文档知识”等。
  - 前台用户在新增/编辑资源时可以内联创建新类别（名称唯一，系统为其分配颜色）。
  - 管理员在后台可以对类别进行完整维护（修改名称/颜色，删除类别）。
  - 删除类别不会删除资源，只会将相关资源的类别字段置为空。
- **标签（Tag）**：
  - 用于对资源打多维度的标签，如“前端”“后端”“文档”“效率”等。
  - 同一资源可以有多个标签，标签可以被多个资源复用。
  - 系统维护标签使用次数统计，支持管理员在后台删除不再需要的标签。
  - 删除标签只会移除资源与该标签的关联，资源本身不受影响。

#### 3.4 收藏与访问历史

- 收藏：
  - 登录用户可以收藏任意可见资源，在卡片或列表中通过心形按钮快速切换收藏状态。
  - “我的收藏”视图只展示当前用户收藏的资源。
- 访问历史：
  - 登录用户点击资源链接时，会记录一条访问历史。
  - 系统会为每个用户保留最多 200 条访问记录，超过时自动删除最旧记录。
  - “最近访问”视图按时间倒序展示用户最近访问过的资源。
- 我创建的：
  - “我创建的”视图展示当前用户作为 owner 创建的所有资源。
- 上述三个视图可在首页通过快速访问入口进行切换，与类别/标签筛选互斥。

#### 3.5 后台管理

- 仅管理员可以访问后台管理页。
- 后台主要模块：
  - 类别管理：查看、创建、编辑和删除类别。
  - 标签管理：查看所有标签及使用次数，删除不需要的标签。
  - 用户管理：查看用户列表，新增用户，修改用户信息（角色、状态等），重置用户密码，删除用户。
  - 系统配置：配置站点标题、副标题、Logo、登录/注册策略、token 过期时间等。
  - 邮件服务配置：配置 SMTP 服务地址、端口、加密方式、发件人信息和认证信息，并支持发送测试邮件。

### 4. 非功能性需求

#### 4.1 安全性

- 所有密码以哈希方式存储（使用安全的哈希算法），严禁明文保存。
- 所有受保护接口通过 JWT 进行身份校验，管理员接口需额外校验角色。
- 登录、忘记密码等接口在错误情况下不暴露“用户是否存在”这一信息，防止账号枚举。
- 邮件配置中的敏感字段（如 SMTP 密码）不会在接口响应中以明文返回。

#### 4.2 可用性与体验

- 桌面端优先设计，保证在典型办公分辨率下拥有良好的阅读和操作体验。
- 在窄屏（如笔记本或小窗口）下，需保证主要页面仍可浏览、筛选资源以及完成后台维护操作。
- 关键操作需有明确反馈：
  - 成功/失败提供 Toast 提示。
  - 删除等危险操作需二次确认。
  - 表单提交中的 loading 状态应阻止重复提交。

#### 4.3 性能与规模假设

- 系统基于单机 SQLite 数据库，面向中小规模资源管理场景部署，预期数据量中小规模：
  - 资源数量：通常在几百到几千条以内。
  - 并发访问量：以小团队、中小组织或单实例使用为主，而非公网高并发场景。
- 请求响应时间在正常网络环境下应保持流畅，首次加载后常用数据可通过前端缓存减少请求次数。
